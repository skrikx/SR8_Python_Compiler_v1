# Governance

SR8 governance is intentionally lightweight and execution-focused.

## Maintainers

Current maintainers are listed in [MAINTAINERS.md](MAINTAINERS.md).

## Decision Model

- Small changes: maintainer merge after review and passing checks.
- Behavioral or architectural changes: documented proposal in issue or PR discussion.
- Release-impacting changes: require at least one maintainer approval and green CI.

## Roadmap Changes

Roadmap priorities are tracked in [ROADMAP.md](ROADMAP.md).
Roadmap updates are accepted when:

- problem statement is clear
- user impact is justified
- scope and non-goals are explicit
- maintenance cost is acceptable

## Escalation Paths

- Bugs and feature requests: issue templates
- Security concerns: [SECURITY.md](SECURITY.md)
- Support questions: [SUPPORT.md](SUPPORT.md)

## Repository Standards

- Keep SR8 local-first.
- Avoid claims that exceed implemented behavior.
- Keep CI, type checks, lint, and tests passing for merged changes.
